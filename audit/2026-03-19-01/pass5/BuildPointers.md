## Correctness Verification

### 1. Does every named item do what it claims?

#### `addressConstantString` (line 22)

**Claim:** Generates a Solidity constant declaration string for a deployed address.

**Verification:** The function produces a string containing `address constant DEPLOYED_ADDRESS = address(<hex>);` with NatSpec. Confirmed by cross-referencing with generated pointer files (e.g., `src/generated/StoxReceipt.pointers.sol` line 17 matches the expected format). The `vm.toString(addr)` call correctly converts an address to its checksummed hex string. The output NatSpec states "The deterministic deploy address of the contract when deployed via the Zoltu factory" which accurately describes the value.

**Result:** Correct.

#### `buildContractPointers` (line 33)

**Claim:** Deploys a contract via Zoltu and generates a pointer file with address, creation code, and runtime code constants.

**Verification:**
1. Line 34: `LibRainDeploy.deployZoltu(creationCode)` deploys via the Zoltu factory and returns the deterministic address. The `deployZoltu` implementation (LibRainDeploy.sol line 154) validates success, non-zero address, and non-empty code.
2. Lines 36-49: `LibFs.buildFileForContract` writes a file at `src/generated/<name>.pointers.sol` containing:
   - `BYTECODE_HASH` (generated by `LibCodeGen.bytecodeHashConstantString` which uses `extcodehash`)
   - `DEPLOYED_ADDRESS` (generated by `addressConstantString`)
   - `CREATION_CODE` (the input `creationCode` bytes, via `LibCodeGen.bytesConstantString`)
   - `RUNTIME_CODE` (`deployed.code`, via `LibCodeGen.bytesConstantString`)
3. The NatSpec strings embedded in the generated output ("The creation bytecode of the contract", "The runtime bytecode of the contract") accurately describe what is stored.

**Result:** Correct. The function faithfully captures all three artifacts (address, creation code, runtime code) plus the codehash.

#### `run` (line 52)

**Claim:** Script entrypoint that generates pointer files for all 7 contracts in dependency order.

**Verification of ordering:**

The Zoltu factory deploys using CREATE (not CREATE2), meaning the deployed address depends on the factory's nonce, which increments with each deployment. The pointer files record these addresses. Some contracts embed pointer file constants in their own creation code via `LibProdDeployV2`:

- `StoxWrappedTokenVaultBeacon` references `LibProdDeployV2.STOX_WRAPPED_TOKEN_VAULT` (from `StoxWrappedTokenVault.pointers.sol`)
- `StoxWrappedTokenVaultBeaconSetDeployer` references `LibProdDeployV2.STOX_WRAPPED_TOKEN_VAULT_BEACON` (from `StoxWrappedTokenVaultBeacon.pointers.sol`)
- `StoxOffchainAssetReceiptVaultBeaconSetDeployer` references `LibProdDeployV2.STOX_RECEIPT` and `LibProdDeployV2.STOX_RECEIPT_VAULT` (from their pointer files)
- `StoxUnifiedDeployer` references `LibProdDeployV2.STOX_OFFCHAIN_ASSET_RECEIPT_VAULT_BEACON_SET_DEPLOYER` and `LibProdDeployV2.STOX_WRAPPED_TOKEN_VAULT_BEACON_SET_DEPLOYER`

The build order in `run()` (lines 55-69) is:
1. `StoxReceipt` (no pointer dependencies)
2. `StoxReceiptVault` (no pointer dependencies)
3. `StoxWrappedTokenVault` (no pointer dependencies)
4. `StoxWrappedTokenVaultBeacon` (depends on StoxWrappedTokenVault pointer -- satisfied by step 3)
5. `StoxWrappedTokenVaultBeaconSetDeployer` (depends on StoxWrappedTokenVaultBeacon pointer -- satisfied by step 4)
6. `StoxOffchainAssetReceiptVaultBeaconSetDeployer` (depends on StoxReceipt and StoxReceiptVault pointers -- satisfied by steps 1-2)
7. `StoxUnifiedDeployer` (depends on both deployer pointers -- satisfied by steps 5-6)

This ordering is correct and complete. All compile-time dependencies are satisfied before the dependent contract's creation code is captured.

**Important subtlety:** `LibProdDeployV2` imports from ALL pointer files. When Foundry compiles this script, it must resolve all pointer files. This works because the pointer files are committed to the repo (as noted in their headers). The script regenerates them in order, and Foundry recompiles incrementally. The circular dependency is documented in the generated files themselves.

**Result:** Correct. Ordering satisfies all dependency constraints.

### 2. Tests vs claims

The `LibProdDeployV2.t.sol` test file (306 lines, 28 tests) provides comprehensive indirect validation:

- **7 deploy address tests:** Verify that deploying each contract via Zoltu produces the address stored in `LibProdDeployV2`.
- **7 codehash tests:** Verify that fresh-compiled contracts match the stored codehash.
- **7 creation code tests:** Verify that pointer `CREATION_CODE` matches `type(...).creationCode`.
- **7 runtime code tests:** Verify that pointer `RUNTIME_CODE` matches deployed bytecode.
- **7 address consistency tests:** Verify that pointer `DEPLOYED_ADDRESS` matches `LibProdDeployV2` constants (not present in the test file for all -- see finding A01-P5-1 below).

The tests also correctly deploy prerequisite contracts before testing dependent ones (e.g., `testDeployAddressStoxWrappedTokenVaultBeacon` first deploys `StoxWrappedTokenVault` via Zoltu before deploying the beacon, because the beacon's constructor calls into the vault implementation address).

### 3. Algorithms and formulas

No mathematical algorithms. The script is purely procedural: deploy, capture address + bytecode, write to file. The Zoltu factory deployment mechanism is well-understood (CREATE opcode determinism).

### 4. Constants and magic numbers

No magic numbers. All values are derived at script execution time. The only hardcoded value is the `BEACON_INITIAL_OWNER` in `LibProdDeployV2` which is documented as `rainlang.eth` with a Basescan link.

### 5. NatSpec vs implementation

No NatSpec on the contract or its functions (covered by pass 3 finding A01-1 as INFO). The inline comments on lines 58-59 and 64 accurately describe the ordering constraints.

The embedded NatSpec in generated output (`@dev The deterministic deploy address...`, `@dev The creation bytecode...`, `@dev The runtime bytecode...`) accurately describes the constants.

### 6. Error conditions vs triggers

`BuildPointers.sol` defines no custom errors. Error handling is delegated to:
- `LibRainDeploy.deployZoltu`: reverts with `DeployFailed` if the Zoltu factory call fails, returns zero address, or deploys empty code. This is correct and sufficient.
- `LibFs.buildFileForContract`: uses Foundry VM cheatcodes (`vm.writeFile`, `vm.removeFile`) which revert on filesystem errors.

### 7. Interface conformance

The contract inherits `Script` from forge-std and correctly implements the `run()` external function as the script entrypoint. No other interfaces to conform to.
